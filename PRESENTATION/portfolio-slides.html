PADRÃO 01

Controller-driven: configuração como dado

Sem controller ANTI-PADRÃO

N tabelas = N artefatos

Cada nova tabela exige alteração de notebook, pipeline ou script. Mudança de configuração vira commit de código — revisão, deploy, risco.

Onboarding lento de novas fontes
Divergência entre código e estado real
Dependência de autor original
10 commits para 10 novas tabelas
Com controller PADRÃO APLICADO

N tabelas = N linhas

Tabelas Delta na camada <catalog>.controller parametrizam toda a execução. Inserir uma linha é incorporar a tabela.

source_to_lakeextração SQL Server → ADLS
landing_to_bronzeconversão Delta
silver_notebooksorquestração com seq
0 commits · 433 tabelas governadas

Governança da pipeline embutida na própria pipeline. Auditoria de mudanças vira histórico de uma tabela Delta. Onboarding de novas fontes vira insert statement, não sprint de desenvolvimento.

Metadata-driven Auditável Escalável

PADRÃO 02

Dynamic Partition Overwrite: quando MERGE não serve

O PROBLEMA

Duplicatas legítimas quebram MERGE

Arquivo ContratadosRC.txt contém linhas 100% idênticas por design: o mesmo funcionário pode aparecer até 21 vezes na mesma competência por múltiplos vínculos.

-- MERGE falha silenciosamente: MERGE INTO fato_contratados AS t USING bronze_contratados AS s ON t.pk = s.pk WHEN MATCHED THEN UPDATE SET * -- Erro: "Cannot perform a MERGE operation when multiple source rows match the same target row"

A SOLUÇÃO

Reescrever a partição inteira

Dynamic Partition Overwrite preserva a semântica: a partição de competência é idêntica à fonte. Duplicatas mantidas, outras partições intactas.

# Configuração Spark: spark.conf.set( "spark.sql.sources. partitionOverwriteMode", "dynamic" ) # Escrita: df.write.format("delta") \ .mode("overwrite") \ .option("partitionOverwriteMode", "dynamic") \ .partitionBy("partition_date") \ .saveAsTable("refined.fato")

OVERWRITE total

Simples mas perigoso

Janela de inconsistência + perda de histórico não presente na fonte atual.

MERGE ACID

Ideal com chaves únicas

Atomicidade linha-a-linha, zero janela. Exige chaves estáveis não-duplicadas.

DPO (dynamic)

Ideal com duplicatas legítimas

Semântica por partição. Preserva histórico fora da janela reescrita.

A escolha de estratégia de escrita não é técnica pura — é semântica de negócio. Entender se a duplicata é erro ou design economiza semanas de debugging.

DPO MERGE ACID Contexto > dogma

QUATRO PRINCÍPIOS

O que eu traria para sua equipe

Documentação é produto

Notas técnicas, case studies e decisões arquiteturais são entregas de primeira classe — não subproduto. A arquitetura viva na cabeça de uma pessoa morre quando ela sai.

Na prática

3 notas técnicas + 4 case studies documentáveis ao final de cada projeto.

Homologação antes do go-live

Toda migração ou refatoração de dados precisa de paridade verificável — não estimativa. Amostragem cria lacuna de confiança que nunca fecha.

Na prática

4,1M PKs validadas em um projeto — 0 bytes copiados entre ambientes.

Automação como multiplicador

CLIs, templates e scripts de deploy são investimentos com retorno composto. Cada diagnóstico manual salvo é hora recuperada para problemas reais.

Na prática

CLI sc reduziu diagnósticos de 15 min para 30 segundos.

Padrões reduzem carga cognitiva

Templates de notebook, estrutura de camadas e convenções de auditoria liberam a mente para o problema real — não para reinventar scaffold.

Na prática

Onboarding de novos notebooks: 2-3 horas → 20-30 minutos com template.
